# PC 데스크톱 위젯 — 설계 (Design)

**상태:** 제안 (구현 전 설계 리뷰)
**목표:** 브라우저(웹앱)가 **열려 있지 않아도** 윈도우 바탕화면에 상주하는 위젯. 필요하면 핀으로 항상-위(always-on-top) 고정. 해빗 / 타임블록 / 타이머 3종.
**전제(검증 완료):** 인앱 타이머(#142), 구글 토큰 자동 갱신(#141), 알림 시스템(#147)은 **이미 구현·배포됨**. 위젯은 이 기능들을 *재사용*하며 재발명하지 않는다.

---

## 0. 왜 웹앱만으로는 불가능한가 (방식 결정의 근거)

| 방식 | 앱 미실행 상태 상주 | 핀(항상-위) | 윈도우 시작 시 자동실행 | 단일 HTML·무빌드 유지 |
|---|---|---|---|---|
| Document Picture-in-Picture | ❌ (탭 필요) | △ | ❌ | ✅ |
| PWA 별도 창 | ❌ (앱 실행 필요) | △(OS 수동) | ❌ | ✅ |
| Windows 11 위젯 보드(PWA) | △(보드 안, 바탕화면 X) | ❌ | — | ❌(MSIX) |
| **Tauri 데스크톱 헬퍼** | ✅ | ✅ | ✅ | ❌(별도 산출물) |

→ 요구사항("앱 안 열려도 바탕화면, 핀 고정, 부팅 시 자동")을 만족하는 건 **독립 데스크톱 앱**뿐이다.

### 0.1 Tauri vs Electron

| | Tauri | Electron |
|---|---|---|
| 바이너리 크기 | ~3–10MB (시스템 WebView2 사용) | ~120MB+ (Chromium 번들) |
| 메모리 | 낮음 | 높음 |
| 프론트엔드 | HTML/CSS/JS (동일 스킬·테마 재사용) | HTML/CSS/JS |
| 셸 언어 | Rust | Node.js |
| 항상-위/투명/프레임리스/트레이/자동시작 | 플러그인 지원 | 지원 |
| 윈도우 WebView2 | Win10/11 기본 내장 | 불필요 |

**결정: Tauri.** 위젯은 "작고 가볍고 항상 떠 있는" 성격이라 메모리/크기가 중요하고, Tauri는 프론트엔드를 기존 웹 기술로 짜므로 테마 토큰과 데이터 스키마를 그대로 재사용할 수 있다. (Electron은 백업안.)

## 1. 아키텍처 개요

```
┌─────────────────────────────┐         ┌──────────────────────────┐
│  웹앱 (index.html, GH Pages) │         │  Tauri 위젯 앱 (윈도우)    │
│  - 데이터 입력/관리 주체      │         │  - 바탕화면 상주 표시기    │
│  - setReturnStorageItem 경로 │         │  - 프레임리스/투명/항상-위 │
│  - Firebase 동기화           │         │  - 트레이 + 자동시작       │
└──────────────┬──────────────┘         └────────────┬─────────────┘
               │ 읽기/쓰기                            │ 읽기(+제한적 쓰기)
               ▼                                      ▼
        ┌──────────────────────────────────────────────────┐
        │   Firebase Firestore  (canonical, 양쪽 공유)       │
        │   users/{uid}/entities/{collection}/items/{_eid}  │
        └──────────────────────────────────────────────────┘
```

- **단일 진실원천은 기존 그대로 Firestore.** 위젯은 동기화 피어를 새로 만들지 않고, 웹앱이 쓰는 같은 Firestore를 본다.
- 위젯은 **자체적으로** Firebase JS SDK로 로그인·구독하므로 웹앱이 꺼져 있어도 최신 데이터를 본다.
- 위젯은 기본 **읽기 전용**으로 시작하고, 쓰기(해빗 체크/타이머 제어)는 단계적으로 추가한다(§5).

### 1.1 데이터 경로 (검증된 기존 스키마 재사용)

| 위젯 | 소스 (localStorage 키 / Firestore collection) | 기존 로직 참조 |
|---|---|---|
| 해빗 | `routine_habits_v1`, `routine_logs_v1` / `routines` | `routineTodayLog()`, `routineStateMeta()` |
| 타임블록 | `task_items_v1`(timeStart/timeEnd), 시간표, 마감 / `tasks` | `renderHomeTimeBlocks()` 의 배치 규칙 |
| 타이머 | `focus_timer_cfg_v1`, `focus_timer_log_v1` / 설정 | `focusTimerState`, `focusTimerStart()` |

위젯 UI는 index.html의 렌더 함수를 직접 호출하지 못한다(전역 스코프·앱 결합). 대신 **데이터 스키마와 테마 토큰만 공유**하고, 위젯 전용 경량 렌더러를 둔다 → "공유 스키마, 분리 UI" 원칙.

### 1.2 인증

- 위젯 WebView 안에서 기존과 동일한 Google 로그인(Firebase Auth `signInWithPopup`/리다이렉트). 토큰은 OS 보안 저장(또는 Tauri store)에 영속.
- 60분 토큰 만료는 Firebase SDK가 자동 갱신(웹앱 #141과 동일 메커니즘) → 위젯도 재로그인 불필요.

## 2. 창/OS 동작

- **프레임리스 + 투명 배경** → 바탕화면 위젯 느낌(둥근 카드).
- **핀 버튼(📌)** → `set_always_on_top` 토글. 끄면 일반 창처럼 다른 창 뒤로.
- **드래그 이동** + 크기 조절, **위치/크기 영속**(다음 부팅 시 복원).
- **시스템 트레이**: 위젯 표시/숨김, 위젯 추가, 종료.
- **자동시작**: `tauri-plugin-autostart` 로 윈도우 로그인 시 자동 실행(설정에서 끄기 가능).
- **다중 위젯**: 위젯 종류별로 독립 창(해빗/타임블록/타이머를 각각 띄우거나 조합). 1차는 단일 창에 탭/스택, 이후 다중 창.

## 3. 위젯 종류별 레이아웃 (1차)

### 3.1 해빗 위젯
- 오늘 습관 목록 + 상태 점(done/skip/rest). 강도(Mini/Plus/Max) 뱃지.
- 완료 비율 헤더(예: `3/7`).
- (쓰기 단계) 항목 탭 → 상태 토글.

### 3.2 타임블록 위젯
- 세로 타임라인, **현재 시각 줄** 강조, "지금 / 다음" 블록을 눈에 띄게.
- 종일/마감 칩 상단.
- 핵심 목적: "지금 뭐 해야 하지?"를 앱 안 열고 즉시 확인.

### 3.3 타이머 위젯
- 기존 focusTimer 상태 미러: 모드(포모도로/카운트다운/스톱워치), 현재 페이즈, 남은/경과 시간, 진행 바.
- (쓰기 단계) 시작/일시정지/리셋 제어 → Firestore 통해 웹앱과 상태 공유.

## 4. 인앱 위젯 설정 (웹앱 측)

웹앱 설정에 **"위젯"** 패널 신설. 위젯 환경설정을 **동기화 키 `widget_prefs_v1`** 에 저장 → Tauri 위젯이 읽어 반영.

설정 항목(초안):
- 표시할 위젯 종류 on/off (해빗/타임블록/타이머)
- 새로고침 주기, 테마(앱 테마 따라가기/고정)
- 타임블록 시간 범위, 해빗 표시 개수
- 위젯 다운로드 링크 + 사용법 안내

> 기존 알림 패널(#147)이 `notif_prefs_v1`를 쓰는 패턴을 그대로 따른다(발명 금지).

## 5. 쓰기(write-back) — 동기화 안전 규칙 ★중요

CLAUDE.md/Stage 6의 핵심: 동기화는 이 repo 최대 리스크. 위젯이 Firestore에 쓰면 웹앱의 `setReturnStorageItem` 부수효과(머지/타임스탬프)를 우회하므로 **반드시 Stage 6 엔티티 계약을 그대로 지켜야** 한다.

- 위젯 쓰기는 **per-entity 문서**(`users/{uid}/entities/{collection}/items/{_eid}`)에 **`updatedAt = now` 스탬프**로 기록 → 웹앱이 평소 LWW 머지로 흡수.
- 삭제는 다루지 않는다(위젯은 생성/삭제 없음, 상태 토글·타이머만) → tombstone 충돌 회피.
- `_eid` 없는 기존 항목은 웹앱이 백필하므로, 위젯은 **이미 `_eid` 있는 항목만** 수정.
- 1차 릴리스는 **읽기 전용**으로 안전하게 출발, 쓰기는 §6 Phase 4+ 에서 검증과 함께 도입.

## 6. 단계 계획 (각 단계 독립 검증·되돌리기 가능)

| Phase | 범위 | 검증 수단 |
|---|---|---|
| **W0** | `/widget` 에 Tauri 프로젝트 스캐폴드(Windows). 프레임리스·투명·always-on-top "hello" 창. | 빌드·실행·핀 토글 동작 |
| **W1** | Firebase Auth 로그인 + Firestore 구독 → **해빗 위젯(읽기)** MVP. | 웹앱에서 습관 체크 → 위젯 반영 |
| **W2** | 핀 버튼, 드래그/리사이즈, 위치 영속, 트레이, 자동시작. | 재부팅 후 위치/자동실행 확인 |
| **W3** | **타임블록 위젯(읽기)** — 현재/다음 강조. | 시간대별 블록·now 라인 정확성 |
| **W4** | **타이머 위젯** — focusTimer 미러 + (쓰기) 시작/정지 제어, §5 규칙 준수. | 위젯↔웹앱 타이머 상태 동기화 |
| **W5** | 웹앱 **위젯 설정 패널**(`widget_prefs_v1`) + 테마 연동. | 설정 변경 → 위젯 반영 |
| **W6** | 해빗 상태 쓰기(§5) + 진단 로그/콘솔. | 두 디바이스 매트릭스(아래) |

각 Phase는 별도 draft PR. **W4(쓰기 도입)와 W6(해빗 쓰기)** 에서 동기화 검증 후 진행.

### 6.1 위젯 쓰기 검증 매트릭스 (W4/W6 병합 전 필수)
1. 위젯에서 해빗 체크 → 웹앱(다른 기기)에 반영, 롤백 없음
2. 웹앱에서 체크 → 위젯 반영(self-echo 손실 없음)
3. 위젯·웹앱 동시 수정 → 최신 `updatedAt` 승리, 다른 쪽 손실 없음
4. 위젯에서 타이머 시작 → 웹앱 타이머 상태 일치
5. 오프라인 위젯 → 재연결 시 정상 머지(부활/유실 없음)

## 7. repo 구조 / 무빌드 철학과의 관계 ★주의

- 웹앱(index.html)은 **계속 단일 파일·무빌드·GH Pages** 유지 — 위젯이 이를 건드리지 않는다.
- 위젯은 **별도 빌드 산출물**: `/widget` 하위 Tauri 프로젝트(자체 `package.json`/`Cargo.toml`). 웹앱 배포 파이프라인과 무관.
- 즉 "이 repo는 무빌드"라는 기존 원칙은 **웹앱에 한해** 유지되고, 위젯은 의도적으로 예외임을 명시.
- 테스트: 위젯의 데이터 어댑터(스키마 매핑)는 기존 `tests/*.test.js` 패턴으로 단위 검증 가능. 창/OS 동작은 수동 검증.

---

## 8. 모바일(iOS/Android)은 별도 경로 (재확인)

- 정적 웹앱은 OS 위젯 API 접근 불가 → 네이티브 앱 없이는 모바일 위젯 불가.
- **우회: 기존 구글 캘린더 동기화 → 구글 캘린더 OS 위젯** 사용.
- 이 우회의 신뢰성은 "앱 꺼진 상태 백그라운드 양방향 동기화"에 달려 있음 → **별도 미해결 항목**(아래 §9).

## 9. 명시적으로 이 설계에 없는 것 / 미해결

- **앱 꺼진 상태 백그라운드 동기화 검증** — 웹앱은 백그라운드 실행이 근본적으로 제한적(서비스워커 한계). 모바일 GCal 우회의 전제이므로 별도 조사 필요. 위젯(PC)은 자체 상주 앱이라 이 문제와 무관.
- 위젯 생성/삭제(데이터 입력)는 범위 밖 — 입력은 웹앱이 담당, 위젯은 표시·간단 토글·타이머 제어만.
- macOS/Linux 위젯 — 1차는 윈도우만. Tauri는 크로스 플랫폼이라 이후 확장 가능.

## 10. 리뷰가 필요한 결정들 (구현 전 확인)

1. **W0 시작 승인** — Tauri로 확정하고 스캐폴드 시작해도 되는가?
2. **다중 창 vs 단일 창** — 1차를 위젯 종류별 독립 창으로? 아니면 한 창에 모아서?
3. **쓰기 도입 시점** — 1차는 읽기 전용으로 충분한가, 아니면 해빗 체크는 처음부터 필요한가?
4. **배포 방식** — 설치형(.msi) + 수동 업데이트로 시작? (Tauri 자동 업데이터는 이후)
