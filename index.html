# Return — Round 5: Evidence-based fixes ## 무엇이 들어갔는가 진단 라운드에서 수집한 콘솔 증거를 바탕으로 3개 root cause를 확정하고 각각에 **최소·안전·되돌리기 가능**한 수정을 적용했습니다. 진단 인프라(Ctrl+Shift+D 패널)는 그대로 유지. ## 진단으로 확정된 root cause ### E1. `_initSettings` 중단 (Cannot read 'groq') - **증거**: 콘솔에 `settings init: TypeError: Cannot read properties of undefined (reading 'groq') at loadApiKeys (...:16259:13) at _initSettings (...:18717:5)` - **인과**: `_initSettings`가 단일 try/catch로 감싸여 있어, `loadApiKeys` throw 시 후속 코드(`fillFbInputs`, `loadDiaryNotionConfig`, **`initFirebase()`** 포함)가 모두 무산 - **영향**: settings 페이지 진입 시 Firebase 자동 init이 안 됨 → 로그인 안 되어 보임 → 사용자 동기화 실패 호소 ### E2. localStorage quota exceeded (diary_entries_v1) - **증거**: 콘솔에 `localStorage.setItem failed for diary_entries_v1 Failed to execute 'setItem' on 'Storage': Setting the value of 'diary_entries_v1' exceeded the quota` × 13회 - **인과**: 다이어리 이미지가 base64로 인라인 저장 → 5MB 누적 한도 도달 → setItem 실패. 메모리 모듈 변수만 갱신, localStorage는 stale. 후속 `fbCollectData`가 stale을 push. - **영향**: 다른 기기 변경이 무시되고 stale로 덮어쓰기. 새로고침해도 stale 그대로. **사용자가 본 시나리오 2의 직접 원인**. ### E6. Notion pull이 Return의 sections 무조건 덮어쓰기 - **증거**: 콘솔이 아니라 사용자 직접 보고 — "노션에서 편집한거 반영이 안되어 있었고, 이후 일기로 그 날짜에 뭐 추가했었는데 동기화 버튼 클릭후 일기탭 기록, 이미지 사라짐" - **인과**: `_pullDiaryPageBody`가 `entry[k] = sectionTexts[k].join('\n\n')`로 무조건 덮어씀. 노션에 sleep/morning/etc가 비어 있으면 `sectionTexts[k]=[]` → 빈 문자열로 Return의 텍스트 덮어쓰기. - **영향**: 한쪽이 비면 다른 쪽이 통째로 사라지는 단방향 sync 효과. ## 적용된 수정 ### Fix #1 — `_initSettings` 단계별 격리 **위치**: `function _initSettings()` (~line 19170) **변경**: 단일 try/catch → 단계별 배열로 풀어 각 step을 자체 try/catch에 넣음. 라우터(`goPage`)의 inits 매핑과 동일한 패턴. ```js var steps = [ ['renderThemeSwatches', function(){ ... }], ['loadApiKeys', function(){ ... }], ... ['initFirebase', function(){ if (fbConfig().apiKey) initFirebase().catch(...); }], ]; steps.forEach(function(step){ try { step[1](); } catch(e){ console.warn('settings init step failed:', step[0], e&&e.message); ... } }); ``` 추가로 `loadApiKeys` 자체도 KEY_MAP 부재에 대해 defensive (fallback map 사용). 어떤 시나리오에서도 throw 안 함. **되돌리는 법**: steps 배열을 다시 일직선 호출 시퀀스로 되돌리면 됨. ### Fix #2 — localStorage quota recovery **위치**: `localStorage.setItem` wrapper 직전 (~line 17191) + `fbCollectData` (~line 16518) + `queueDiaryNotionSave` (~line 17440) 추가된 것: - `window.__staleQuotaKeys` — 키별 stale 표시 (per session) - `__compressDataUrl(dataUrl, maxDim, quality)` — canvas 다운스케일 헬퍼 - `__compressDiaryImagesAndRetry()` — 3패스 점진적 압축 (1600/0.85 → 1024/0.78 → 720/0.72). 각 entry image별로 적용, 중간/끝에 setItem 재시도. 성공하면 stale 해제. - wrapper의 quota error 분기 — diary_entries_v1이면 백그라운드 압축 시작, stale 표시 - `fbCollectData` — stale 키는 push 대상에서 제외 - `queueDiaryNotionSave` — diary stale이면 노션 푸시 건너뜀 **사용자 사용자 결정 반영**: "최대한 수동으로 압축 시도 후 실패하면 표시" → 3패스 압축 후 모두 실패 시 명시적 토스트: > 이미지가 많아 로컬 저장에 실패했어요. 일부 이미지를 삭제해주세요. 이 기기의 일기는 클라우드로 올라가지 않습니다. **핵심 보장**: 압축이 실패해도 **stale 로컬이 cloud를 덮어쓰는 일은 절대 없음**. 최악의 시나리오는 "이 기기에서만 diary가 cloud로 push 안 됨" — "다른 기기 데이터 손실" 아님. **되돌리는 법**: `__staleQuotaKeys`/`__compressDiaryImagesAndRetry`/`__markStaleQuota`/`__clearStaleQuota` 블록 통째로 삭제, wrapper의 quota error catch를 원래대로 `console.warn` 만 하도록 되돌림, `fbCollectData`/`queueDiaryNotionSave`의 stale check 제거. ### Fix #3 — Notion pull은 non-empty일 때만 덮어씀 **위치**: `_pullDiaryPageBody` (~line 18223) **변경**: ```js Object.keys(sectionTexts).forEach(function(k){ var pulled = (sectionTexts[k]||[]).join('\n\n'); if (pulled && pulled.trim().length) { entry[k] = pulled; } /* else: keep entry[k] as-is. */ }); ``` 빈 노션 sections가 채워진 Return sections를 덮을 수 없음. 양방향에서 자연스러운 merge: 둘 다 비면 그대로, 한 쪽만 채워졌으면 그쪽이 살아남음, 둘 다 채워졌으면 노션 우선(사용자 결정한 정책 유지). **되돌리는 법**: 분기 제거, 원래 `entry[k]=sectionTexts[k].join('\n\n')` 한 줄로. ## 진단 패널 갱신 `Ctrl+Shift+D` 패널에 새 라인 추가: ``` STALE keys: diary_entries_v1 ← cloud NOT being overwritten ``` 이 라인이 보이면 그 키는 cloud로 push 안 되고 있다는 명시적 표시. ## 검증 시나리오 (지난번과 동일) ### A. settings 진입 시 Firebase init이 막히지 않는지 (E1) 1. 시크릿창에서 앱 열기, 로그인 2. Settings 탭 진입 3. 콘솔에서 `Cannot read properties of undefined (reading 'groq')`이 떠도 OK 4. **Settings 페이지의 다른 입력란이 채워지는지 확인** (Firebase config 입력란, 노션 input 등) 5. 진단 패널의 `1st fbLoad start` 가 채워지는지 확인 — 채워졌다면 Firebase init이 실행된 것 ### B. 다이어리 quota 시나리오 (E2) 1. 진단 패널 켜기 2. 다이어리에 이미지 여러 개 첨부 (5MB 근접까지) 3. setItem fail 시 새 토스트 떠야 함: "이미지가 너무 많아 로컬 저장 한도에 닿았어요. 압축 시도 중…" 4. 진단 패널의 STALE keys 라인 확인 → `diary_entries_v1` 표시 5. 압축 성공 시 다른 토스트: "이미지 압축으로 저장 공간을 확보했어요" → STALE 라인 사라짐 6. 압축 실패 시 명시적 토스트로 알림 ### C. 두 기기 동시 편집 (E2 후속) 1. A 기기와 B 기기 둘 다 패널 켜기 2. B에서 인박스 항목 추가 → A에서 같은 항목이 ~1초 내에 보여야 함 3. A의 패널에서 `onSnapshot fired` → `fbApplyData ok` 시퀀스 보여야 함 4. **A에 STALE keys가 있다면 그 키만 sync 못 함** — 그 키 외의 변경은 들어와야 함 ### D. Notion 양방향 (E6) 1. 노션에서 어느 날짜에 이미지 추가 2. Return의 같은 날짜에 일기 텍스트 추가 (sleep, morning 등) 3. ⇅ 동기화 클릭 4. 노션 이미지가 Return diary에 들어와야 함 5. Return의 일기 텍스트는 **사라지면 안 됨** — 노션이 비웠어도 보존 6. 푸시 단계에서 노션의 그 날짜에 Return의 텍스트도 올라가야 함 ## 검증 결과 ``` JS_OK div 1111 1111 OK section 29 29 OK main 5 5 OK button 338 338 OK select 16 16 OK details 1 1 OK summary 1 1 OK textarea 20 20 OK form 0 0 OK ``` ## 손대지 않은 것 (의도적) - **E3 deleteItem 인자 누락**: 사용자가 명시한 3개 (E1/E2/E6)에 미포함. 매우 작은 1줄 수정이고 sync와 무관하지만 별도 컨펌 후 처리할 사항. follow-up에서 확인 받아 진행 권장. - **시나리오 4의 네트워크 에러**: 환경 문제이고 코드 버그 아님. 다만 끊긴 동안 큐가 어떻게 처리되는지는 사용자가 다음 진단 라운드에서 패널 snapshot으로 보고하면 확인 가능. - **시나리오 2의 인과 최종 입증**: quota 가설이 가장 강하나 패널 snapshot이 없어 최종 입증은 안 됨. 위 시나리오 C로 확인 가능. - **이전 라운드의 추측 패치(fbApplyData 리로더 리스트 등)**: 진단으로 입증되지 않은 것이라 추가 변경 안 함. 이번 round에 들어간 것은 모두 입증된 또는 사용자가 직접 본 증상. ## 진단 인프라 사용법 (재공유) - `Ctrl+Shift+D` 토글 - 콘솔: `__SYNC_DEBUG.snapshot()` / `.copy()` / `.dump()` - copy 버튼 → 클립보드에 JSON snapshot 전체 복사. 다음 동기화 문제 보고시 그 JSON 붙여 주세요. ## 권장 follow-up 1. 위 검증 시나리오 A~D 실행 후 결과 보고 2. 시나리오 C에서 새 증상이 발견되면 패널 snapshot 첨부 3.
