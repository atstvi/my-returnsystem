# 프롬프트 템플릿 모음

다른 세션에서도 바로 복사해서 쓰는 작업 지시 템플릿입니다.
대괄호 `[...]` 안만 채우면 됩니다. 모르는 항목은 비워두고 "모름"이라고 적어도 됩니다.

---

## 🐛 버그 수정용 프롬프트

```
## 목적
[버그명]: [한 줄 설명]
이 버그 하나만 이번 턴에 완전히 해결해줘. 다른 작업과 병행하지 말 것 (한 번에 하나씩).

## 맥락
- 앱/기능: [앱 이름 / 해당 기능]
- 증상: [내가 눈으로 보는 현상을 있는 그대로]
- 발생 조건: [어떤 행동을 했을 때 / 항상인지 가끔인지]
- 최근 변경사항: [이 버그가 언제부터인지 / 최근 바꾼 것 / 모르면 "모름"]

## 입력 조건 (역질문 유도)
나는 어떤 정보가 원인 분석에 필요한지, 무엇이 작업에 영향을 주는 요인인지 모를 때가 많아.
그러니 아래 항목 중 이 버그를 진단하는 데 필요한 게 있으면 작업 시작 전에 먼저 물어봐줘.
내가 모르면 모른다고 답할게. 그리고 아래에 없더라도 결정적으로 필요한 정보가 있으면 추가로 질문해줘:
- 브라우저 / 기기 종류
- 로그인 상태 / 특정 사용자에서만 재현되는지
- 네트워크 환경 (오프라인/온라인)
- 재현 빈도 (항상 / 가끔 / 특정 조건일 때만)
- 관련 설정값 (예: API 키, 기능 On/Off)
- [이 앱·기능에서 추가로 필요할 것 같은 항목을 너가 판단해서 질문]

## 작업 규칙 (반드시 지킬 것)
1. 추측 절대 금지
   - 고치기 전에 관련 전체 코드를 실제로 읽어. 원인이 코드에서 확인된 것만 수정.
   - "아마 이거일 것이다" 식으로 손대지 말 것.
2. 진단 먼저 (불확실할 때)
   - 원인이 코드만 봐서 확실하지 않으면, 먼저 진단 수단을 추가해서 실제 런타임 데이터를 확보한 뒤 고쳐.
   - 진단 수단 예시: 임시 콘솔 로그(F12로 확인), 추적 기록(trace) 버퍼, 화면에 결과를 찍는 진단 버튼.
   - 진단으로 원인을 "확정"한 다음 수정에 들어갈 것.
3. 자동 연관 수정 (당연한 건 당연히 같이)
   - 원인을 찾으면, 같이 고쳐야 당연한 연관 부분도 알아서 함께 수정해. 일일이 지시하지 않을게.
   - 매번 사용자에게 사소한 사항을 묻지 말고 시중 앱을 참고하여 프로젝트의 표준과 최선의 개발 관행(Best Practices)에 따라 주도적      으로 작업
   - 예: 한쪽 방향만 고치면 반대 방향도 깨지는지, 같은 패턴의 다른 호출부, 동기화/저장/에러 처리/UI 반영.
4. 루프 검증 (적당한 횟수, 무한 금지)
   - 수정 후 내가 확인하기 전에 너가 스스로 코드 재검토 / 테스트 / 진단 결과로 검증해.
   - 아직 문제가 있으면 원인을 재분석하고 다시 수정. "이 정도면 됐겠지" 하고 멈추지 말 것.
   - 단, 토큰을 고려해 최대 3회까지만 반복. 3회 후에도 해결 안 되면 멈추고
     현재 상태 + 막힌 이유 + 다음에 필요한 정보를 정확히 보고할 것.
5. 완료 기준
   - 수정 → 테스트 통과 → 배포(커밋/푸시)까지 끝난 상태.
   - 마지막에 "무엇이 원인이었고, 어떻게 고쳤고, 어떻게 확인했는지" 한 단락으로 정리.

## 역질문 요청
시작 전에: 위 입력 조건 중 빠진 것 또는 원인 분석에 결정적인 정보가 더 필요하면 먼저 질문해줘.
작업 중에: 원인이 두 갈래로 갈리거나 구조적으로 큰 결정이 필요하면, 임의로 고르지 말고 선택지를 제시하고 물어봐줘.
```

---

## ✨ 새 기능 / 수정 요청용 프롬프트

```
## 목적
[기능명]: [한 줄 설명]
이 작업 하나만 이번 턴에 완성해줘. 다른 기능과 병행하지 말 것 (한 번에 하나씩).

## 맥락
- 앱/기능: [앱 이름 / 관련 기능]
- 현재 상태: [지금 어떻게 동작하는지]
- 원하는 상태: [수정 후 어떻게 동작해야 하는지 — 최대한 구체적으로]
- 이유/배경: [왜 이 변경이 필요한지]
- 관련 기능: [이 변경이 영향을 줄 수 있는 다른 기능 / 모르면 "모름"]

## 입력 조건 (역질문 유도)
나는 어떤 결정이 구현 방향에 영향을 주는지 모를 때가 많아.
그러니 아래 항목 중 구현 방향을 가르는 게 있으면 시작 전에 먼저 물어봐줘.
선택지를 2~3개 제시하고 각각의 트레이드오프를 짧게 설명해주면 더 좋아.
아래에 없더라도 결정적인 선택지가 있으면 추가로 질문해줘:
- 우선순위: 속도 vs 정확도 / 단순함 vs 완성도
- 엣지 케이스 처리 방향 (값이 없을 때 / 에러일 때 어떻게)
- UI 변경 허용 여부 (기존 화면을 바꿔도 되는지)
- 하위 호환 필요 여부 (기존 데이터·설정 유지해야 하는지)
- 성능 제약 (API 호출 횟수 / 응답 속도 목표)
- [이 앱·기능에서 추가로 필요할 것 같은 항목을 너가 판단해서 질문]

## 작업 규칙 (반드시 지킬 것)
1. 추측 절대 금지
   - 구현 전에 기존 코드의 연관 부분을 전부 실제로 읽어.
   - 비슷한 기능·패턴이 이미 있으면 그 스타일과 구조를 따를 것. 새로 발명하지 말 것.
2. 자동 연관 수정 (당연한 건 당연히 같이)
   - 이 기능을 넣으면 당연히 같이 바꿔야 하는 부분도 알아서 포함해. 일일이 지시하지 않을게.
   - 매번 사용자에게 사소한 사항을 묻지 말고 시중 앱을 참고하여 프로젝트의 표준과 최선의 개발 관행(Best Practices)에 따라 주도적      으로 작업
   - 예: 설정 저장/불러오기, UI 반영, 에러 처리, 동기화, 기존 데이터 마이그레이션.
3. 동작 확인 수단 포함
   - 이 기능이 실제로 작동하는지 내가/너가 확인할 수 있는 수단을 함께 만들어.
   - 예시: 진단 버튼, 콘솔 로그(F12 확인), 테스트 케이스.
4. 루프 검증 (적당한 횟수, 무한 금지)
   - 구현 후 내가 확인하기 전에 너가 스스로 코드 재검토 / 테스트 / 문법 검사 / 진단 결과로 검증해.
   - 의도대로 안 되면 원인을 재분석하고 다시 구현. "이 정도면 됐겠지" 하고 멈추지 말 것.
   - 단, 토큰을 고려해 최대 3회까지만 반복. 3회 후에도 안 되면 멈추고
     현재 상태 + 막힌 이유 + 다음에 필요한 정보를 정확히 보고할 것.
5. 완료 기준
   - 구현 → 테스트 통과 → 배포(커밋/푸시)까지 끝난 상태.
   - 마지막에 "이제 이렇게 쓰면 됩니다" 사용법 한 줄 + 무엇을 어떻게 확인했는지 요약.

## 역질문 요청
시작 전에: 위 입력 조건 중 빠진 것 또는 구현 선택에 결정적인 정보가 더 필요하면 먼저 질문해줘.
작업 중에: 방향이 갈리는 결정이 생기면 임의로 고르지 말고 선택지 + 트레이드오프를 제시하고 물어봐줘.
```

## 이 템플릿이 담고 있는 "성공 요인" (참고)

이전 세션에서 작업 성공률이 높았던 이유를 그대로 규칙으로 박아 넣었습니다:

| 성공 요인 | 템플릿 반영 위치 |
|---|---|
| 진단으로 원인을 "확정"한 뒤 수정 (추측 X) | 작업 규칙 1·2 (추측 금지 + 진단 먼저) |
| 추적 기록(trace)·콘솔 로그로 런타임 데이터 확보 | 작업 규칙 2·3 (진단 수단 / F12 확인) |
| 한 곳 고치면서 연관된 곳(동기화·저장·UI)까지 연달아 수정 | 작업 규칙: 자동 연관 수정 |
| "됐겠지" 대신 결과로 재확인하는 루프 (단, 유한) | 작업 규칙: 루프 검증 (최대 3회) |
| 내가 모르는 필요 정보를 너가 역질문으로 끌어냄 | 입력 조건 + 역질문 요청 |
| 한 번에 하나씩 | 목적 섹션 명시 |
